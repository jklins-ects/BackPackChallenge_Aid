# BackPackChallenge

BackPackChallenge is a Node.js + Express + MongoDB event app for managing participant progress across multiple stations.

It is designed for situations where a large visiting group is split into smaller groups and rotates through stations in any order. Participants can:

- be pre-created in a group
- receive a short participant code
- optionally be linked to an NFC chip
- earn activity stats
- upload or create a logo
- later have all of that tied together into one participant record

## Current Features

- Express REST API for participant management
- MongoDB schema validation and setup scripts
- Bulk creation of participant shell records
- Unique participant codes for each participant
- Group-based participant lookup
- Lookup by participant code
- Lookup by NFC id
- Link an NFC id to an existing participant
- Award stats by participant code
- Award stats by NFC id
- Pending NFC activity events that can be merged later when a participant is linked
- Swagger API documentation
- Sample printable group participant page
- Sample logo creation page that saves cropped logos to MongoDB

## Tech Stack

- Node.js
- Express
- MongoDB
- Swagger / OpenAPI
- HTML / CSS / JavaScript frontend pages
- Canvas for autogenerated logo images

## Data Model Overview

### Participants

A participant record can include:

- `groupId`
- `participantCode`
- `nfcId`
- `firstName`
- `lastName`
- `logo`
- `stats`
- `createdAt`
- `updatedAt`

Example:

```json
{
    "groupId": "group-101",
    "participantCode": "A7K2",
    "nfcId": "04AABBCCDD",
    "firstName": "Jason",
    "lastName": "Smith",
    "logo": "data:image/png;base64,...",
    "stats": {
        "activity1": 0,
        "activity2": 5,
        "activity3": 0,
        "activity4": 10,
        "activity5": 0,
        "activity6": 0,
        "activity7": 0,
        "activity8": 0,
        "activity9": 0
    },
    "createdAt": "2026-03-24T20:00:00.000Z",
    "updatedAt": "2026-03-24T20:15:00.000Z"
}
```

### Activity Events

Activity events are stored separately so scoring actions can be tracked and merged later.

An activity event can include:

- `participantId`
- `groupId`
- `participantCode`
- `nfcId`
- `stationId`
- `eventType`
- `activityKey`
- `points`
- `resolved`
- `resolvedAt`
- `createdAt`

This allows activity scores to be recorded before the participant is fully linked.

## How This Can Work With NFC Chips

This project is built to support an event where students receive chips that initially only identify them by a simple pair of values:

- `groupId`
- `participantCode`

That is the easiest and safest event model because station order does not matter.

### Recommended Chip Strategy

Each student receives:

- a chip or keychain
- a `groupId`
- a short `participantCode`

The chip may eventually contain more information, but the simplest usable starting point is that the system knows how to associate a scan or form entry with:

- the student’s group
- the student’s participant code

### Why This Works Well

Students may visit stations in any order, so you do not want the whole system to depend on registration happening first.

Instead:

- some stations can identify students by typing `groupId` + `participantCode`
- some stations can identify students by NFC
- at any point, those two can be joined together

That means logo updates, stat awards, and registration details can all happen independently and still end up tied to the same participant.

## How Data Can Be Tied Together

This repo supports a workflow where participant information can be joined together later if needed.

### Example Event Flow

A student arrives with:

- `groupId = group-101`
- `participantCode = A7K2`

#### Station A: Online Logo Form

At a logo station, the student can:

- select their `groupId`
- select their participant
- upload or paste an image
- crop it
- save it to MongoDB

This updates the participant’s `logo`.

#### Station B: Online Form

At another station, a form could:

- select `groupId`
- select participant
- update name or other fields

This updates the participant record directly.

#### Station C: NFC-Based Scoring

At an NFC-based station, a chip can be read and stat activity can be associated with that chip.

If the chip is already linked to a participant, the stat can be added directly to that participant.

If the chip is not linked yet, the activity can still be stored as a pending event and later merged into the participant once the student is linked.

### Joining It All Together

At any point, a staff member can:

- read the NFC chip
- open a form
- select the student’s `groupId`
- select the correct participant
- link the chip to that participant

Once linked:

- previously stored NFC activity events can be merged
- future NFC activity can be assigned directly
- online form data and chip-based data now point to the same participant record

This means the student’s full data can be tied together even if stations were visited out of order.

## Suggested Real-World Workflow

A practical event flow could look like this:

### Before the Event

- bulk create participant shells for each visiting group
- print participant code sheets
- hand out chips or keychains

### During the Event

Students rotate through stations in any order.

Some stations may:

- assign stats
- collect names
- collect logos
- read NFC chips

### At Any Time

A helper can:

- read the chip
- select the group
- select the participant
- link them together

After that, all data can be viewed as one participant profile.

## Important Current Limitation

There is **not yet any finished frontend handling for NFC stat writing**.

Specifically:

- the backend structure for activity events and linking is in progress / available for extension
- a sample logo form exists
- a sample group display page exists
- a complete station page that reads an NFC chip and writes stats through the app is **not included yet**

In other words:

- viewing data: sample page included
- updating logos: sample page included
- actual NFC stat-writing station UI: **not yet implemented**
